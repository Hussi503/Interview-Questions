# 🔴1. What is CloudWatch, and how do you monitor CPU, memory, and disk across many EC2 instances?

**Amazon CloudWatch** is AWS's native monitoring and observability service. It collects metrics, logs, and events from AWS resources and applications, helping us monitor infrastructure health, troubleshoot issues, and trigger automated actions.

In production, monitoring hundreds of EC2 instances manually isn't practical, so we centralize monitoring through CloudWatch.

For **CPU utilization**, AWS automatically publishes metrics such as CPUUtilization without any additional configuration.

However, **Memory utilization** and **Disk usage** are operating system metrics, so they are **not collected by default**. We install the **CloudWatch Agent** on every EC2 instance, configure it to collect memory, disk, swap, and filesystem metrics, and send them to CloudWatch.

We then organize servers using tags such as Production, Application, and Environment, and build CloudWatch Dashboards to monitor the entire fleet. CloudWatch Alarms notify our team through SNS whenever thresholds are exceeded.

### Real-Time Example

In one of our production environments, we monitored nearly 150 EC2 instances.

We configured the CloudWatch Agent through Ansible during server provisioning. CloudWatch dashboards displayed CPU, memory, and disk metrics for all application servers. If CPU exceeded **80%**, memory exceeded **85%**, or disk utilization crossed **90%**, CloudWatch automatically triggered an SNS notification, which forwarded alerts to our Slack channel and email distribution list. This allowed the operations team to respond before users experienced service degradation.

### Best Practices

- Install CloudWatch Agent using Ansible or Systems Manager.
- Use CloudWatch Dashboards for centralized monitoring.
- Configure CloudWatch Alarms for critical metrics.
- Organize instances using AWS tags.
- Monitor trends instead of relying only on current values.

# 🔴2. What is Amazon CloudWatch? Difference between CloudWatch Metrics and Logs?

Amazon **CloudWatch** is AWS's monitoring service that provides visibility into infrastructure, applications, and AWS services. It collects metrics, stores logs, generates alarms, and integrates with automation services for operational monitoring.

The main difference between **Metrics** and **Logs** is the type of data they store.

### CloudWatch Metrics

Metrics are **numerical time-series data** collected at regular intervals.

Examples include:

- CPU Utilization
- Network In
- Network Out
- Disk Read Operations
- ALB Request Count
- Lambda Duration

Metrics help answer questions like:

- Is CPU utilization increasing?
- Is latency growing?
- How many requests are hitting the load balancer?

---

### CloudWatch Logs

Logs contain **detailed textual information** generated by operating systems, applications, or AWS services.

Examples include:

- Nginx logs
- Apache logs
- Application logs
- System logs
- Docker logs
- EKS logs

Logs help answer questions like:

- Why did the application fail?
- Which exception occurred?
- Which user generated the error?

---

### Real-Time Example

In one of our production environments:

CloudWatch Metrics showed CPU utilization increasing from **35% to 95%**.

To determine the root cause, we opened CloudWatch Logs and found repeated **Java OutOfMemoryError** exceptions in the application logs.

Metrics identified **that** a problem existed, while logs explained **why** it happened.

### Easy Way to Remember

| Metrics | Logs |
|----------|------|
| Numerical Data | Textual Data |
| Performance Monitoring | Troubleshooting |
| CPU, Memory, Network | Application & System Logs |
| Used for Dashboards & Alarms | Used for Root Cause Analysis |

# 🔴3. How do you monitor EC2 CPU, memory, disk, and ALB 5XX errors?

In production, we monitor infrastructure using a combination of **CloudWatch Metrics**, **CloudWatch Agent**, **CloudWatch Alarms**, and **CloudWatch Dashboards**.

Monitoring setup includes:

- **CPU Utilization** → Native CloudWatch Metric.
- **Memory Utilization** → CloudWatch Agent.
- **Disk Usage** → CloudWatch Agent.
- **ALB HTTPCode_ELB_5XX_Count** and **HTTPCode_Target_5XX_Count** → Native ALB Metrics.

We configure CloudWatch Alarms with threshold values based on application requirements.

Typical thresholds include:

- CPU > 80%
- Memory > 85%
- Disk Usage > 90%
- ALB 5XX Errors > 10 within 5 minutes

Alerts are delivered through **Amazon SNS**, which forwards notifications to Slack, Microsoft Teams, email, or PagerDuty.

### Real-Time Example

During one production deployment, users reported intermittent application failures.

CloudWatch showed:

- CPU Normal
- Memory Normal
- Disk Normal
- ALB 5XX Errors Increasing

We investigated CloudWatch Logs and discovered that one backend application server had failed after deployment.

The unhealthy instance was automatically removed from the ALB Target Group, the deployment was rolled back, and traffic resumed normally.

Because CloudWatch detected the issue immediately, the incident was resolved within minutes.

### Best Practices

- Install CloudWatch Agent.
- Configure CloudWatch Dashboards.
- Monitor ALB Target Health.
- Create meaningful alarm thresholds.
- Integrate alarms with incident management platforms.

# 🔴4. What is the role of CloudTrail?

**AWS CloudTrail** is the auditing and governance service that records every API call made within an AWS account.

Unlike CloudWatch, which focuses on monitoring infrastructure health, CloudTrail answers questions such as:

- Who created an EC2 instance?
- Who deleted an S3 bucket?
- Who modified an IAM policy?
- When was a Security Group changed?
- Which user assumed an IAM role?

Every API request includes:

- User identity
- Timestamp
- Source IP Address
- AWS Service
- API Operation
- Request Parameters
- Response Status

CloudTrail stores these audit logs securely in Amazon S3 and can integrate with CloudWatch Logs for real-time monitoring.

### Real-Time Example

In one of our production environments, an EC2 security group was accidentally modified, exposing SSH access publicly.

Instead of manually investigating, we queried CloudTrail and identified:

- The IAM user who made the change.
- The exact API call (`AuthorizeSecurityGroupIngress`).
- The timestamp.
- The source IP address.

This helped us quickly revert the change and improve IAM policies to prevent similar incidents.

### Best Practices

- Enable CloudTrail in all AWS regions.
- Store logs in encrypted S3 buckets.
- Enable log file validation.
- Integrate CloudTrail with CloudWatch.
- Restrict access to audit logs.

### Easy Way to Remember

**CloudWatch → How is my infrastructure performing?**

**CloudTrail → Who changed my infrastructure?**

# 🔴5. How do you set alerts for production incidents?

In production, alerting is designed to notify the operations team before customers are significantly impacted.

We use **CloudWatch Alarms** to monitor critical infrastructure and application metrics.

Common alerts include:

- High CPU Utilization
- High Memory Usage
- High Disk Usage
- ALB 5XX Errors
- High Response Time
- Low Healthy Host Count
- RDS Storage Nearly Full
- Auto Scaling Failures

Each alarm is associated with an **SNS Topic**, which distributes notifications to:

- Email
- Slack
- Microsoft Teams
- PagerDuty
- Opsgenie

Depending on the severity, alerts may also trigger automatic remediation using **AWS Lambda** or **Systems Manager Automation**.

### Real-Time Example

In one of our production applications:

CloudWatch detected:

- CPU > 85%
- ALB Response Time > 2 seconds

CloudWatch Alarm triggered an SNS notification.

SNS forwarded the alert to Slack and PagerDuty.

At the same time, an Auto Scaling policy launched an additional EC2 instance, reducing CPU utilization and restoring application performance without manual intervention.

### Best Practices

- Alert only on actionable metrics.
- Avoid excessive alerting to reduce alert fatigue.
- Use different thresholds for Warning and Critical alerts.
- Test alarms periodically.
- Integrate alerts with on-call platforms.

# 🔴6. What are EKS Control Plane Logs?

Amazon EKS provides **Control Plane Logs** that help monitor and troubleshoot Kubernetes cluster operations.

These logs are generated by the managed Kubernetes control plane and can be sent directly to **CloudWatch Logs**.

Available Control Plane Logs include:

- **API Server**
  - Records all API requests made to the Kubernetes API Server.

- **Audit**
  - Captures every Kubernetes API request for security and compliance auditing.

- **Authenticator**
  - Records IAM authentication attempts and authorization decisions.

- **Controller Manager**
  - Shows controller activities such as Deployment reconciliation, ReplicaSet management, and Node management.

- **Scheduler**
  - Records Pod scheduling decisions and scheduling failures.

### Real-Time Example

During one production deployment, developers reported that Pods were not being created.

The Scheduler logs showed insufficient CPU resources on worker nodes.

The Controller Manager logs confirmed that the Deployment was continuously attempting to create new Pods.

After reviewing Cluster Autoscaler logs, we found that worker nodes had reached the maximum Auto Scaling limit.

Increasing the node group size resolved the issue.

Without EKS Control Plane Logs, identifying the scheduling problem would have taken significantly longer.

### Best Practices

- Enable all EKS Control Plane Logs for production clusters.
- Send logs to CloudWatch Logs.
- Configure retention policies.
- Monitor authentication failures.
- Review Audit Logs regularly for security compliance.

### Easy Way to Remember

| Control Plane Log | Purpose |
|-------------------|---------|
| API Server | Records Kubernetes API requests |
| Audit | Security and compliance auditing |
| Authenticator | IAM authentication and authorization |
| Controller Manager | Resource reconciliation activities |
| Scheduler | Pod scheduling decisions |
